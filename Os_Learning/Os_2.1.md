# 虚拟化cpu

## cpu的硬件结构

### 1.核心运算控制寄存

* 算术逻辑单元 (ALU - Arithmetic Logic Unit)： 工厂的“车间”。负责所有的加减乘除运算和逻辑判断（如“是否大于零”）。它是 CPU 的“肌肉”。

* 控制单元 (CU - Control Unit)： 工厂的“调度室”。负责从内存抓取指令（Fetch）、解析指令（Decode），并指挥 ALU 和其他组件干活。

* 寄存器组 (Registers)： 工人的“双手”。CPU 内部速度最快、容量最小的存储空间。

    * PC (程序计数器)： 记录下一条指令在哪。

    * IR (指令寄存器)： 存放当前正在翻译的指令。

    * 通用寄存器： 暂时存放计算过程中的操作数。

### 2.现代CPU辅助部门

* MMU (内存管理单元)： “翻译官”。我们在内存虚拟化里讲过，它是负责把“虚拟地址”瞬间翻译成“物理地址”的硬件开关。

* TLB (转换后备缓冲区)： “翻译官的快表”。它是 MMU 内部的一块极速缓存，专门存储最近用过的页表项，解决“查字典太慢”的问题。

* FPU (浮点运算单元)： “高级精算师”。专门处理复杂的浮点数（小数）运算，以前是独立的芯片，现在都集成在核心内部。

* 分支预测器 (Branch Predictor)： “先知”。它猜测你的 if-else 接下来会走哪条路，并提前把那里的指令抓回来（执行流水线优化）。

### 3.多核架构

* 每个核心 (Core)： 拥有自己独立的 ALU、CU、L1 和 L2 Cache。

* 环形总线/网格 (Uncore / Interconnect)： 将这些核心串联在一起。

* L3 Cache： 就像一个巨大的**“共享仓库”**，所有核心都能访问。

* 内存控制器 (IMC)： 直接集成在 CPU 里，负责跟外面的内存条通信。

### 4.数据经过cpu的路线
* CU 根据 PC 指向的地址，通过 总线 从内存拿到指令。

* 指令进入 IR，CU 进行解码。

* 如果需要数据，MMU 查阅 TLB 进行翻译，从 Cache 或内存把数据读入 寄存器。

* ALU 从寄存器拿数，算出结果，再放回寄存器。

* 最后由 总线 写回内存。

## CPU的虚拟化

### 核心矛盾：效率和控制
* 追求效率： 我们希望程序跑在 CPU 上时，能像真正在物理机上跑一样快，所以必须让程序直接执行指令，而不是由 OS 去逐条解释（那样太慢）。

* 追求控制： 如果程序直接在 CPU 上跑，它可能会执行 halt 关机指令，或者读写别人的内存。OS 必须能随时“切断”它的电源。

* 解决方案：权限系统（控制cpu在运算中直接执行，限制cpu执行其他的操作）
    * 用户模式 (User Mode / Ring 3)： 程序（进程）跑在这里。它只能做算术题、读写自己的内存。如果它敢尝试执行“关机”或“修改磁盘”等敏感指令，CPU 硬件会直接抛出一个错误（异常），并把控制权强行交给 OS。

    * 内核模式 (Kernel Mode / Ring 0)： 操作系统跑在这里。它可以执行任何指令，包括操作硬件和修改所有内存。
### 中断

* 物理实现是时钟中断 (Timer Interrupt)：
    * 埋雷： OS 在启动时，命令硬件计时器每隔 $10ms$ 发出一个电信号。
    * 强制打断： 当信号响起，CPU 硬件会无条件地停止当前程序的运行。
    * 触发陷阱： CPU 自动跳转到 OS 预先设定好的陷阱处理程序 (Trap Handler) 中。此时，OS 终于“醒”了过来。它就像一个严厉的班主任，走进教室收走当前学生的卷子（保存现场），然后把位置交给下一个学生。

### 时空转移
* 保存旧人： 将进程 A 当前所有的寄存器状态（包括程序计数器 $PC$、栈指针 $SP$ 等）保存到内存中的一个结构体里（在 Linux 中就是 task_struct）。
* 加载新人： 从内存中读取进程 B 之前存好的寄存器值，强行塞进 CPU 的物理寄存器里。
* 返回用户态： 执行一条特殊的“返回陷阱”指令，CPU 瞬间切换回用户模式。
